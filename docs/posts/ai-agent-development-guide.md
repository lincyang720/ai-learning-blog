# AI Agent 开发完全指南:从基础到实践

## 什么是 AI Agent?

AI Agent(智能代理)是一种能够自主决策、使用工具并完成复杂任务的 AI 系统。与简单的 LLM 调用不同,Agent 能够:

- 🤔 **自主规划**: 动态决定执行步骤
- 🛠️ **使用工具**: 调用外部 API 和服务
- 🔄 **迭代优化**: 根据反馈调整策略
- 🎯 **目标导向**: 持续工作直到完成任务

## Agent vs Workflow: 核心区别

在构建 AI 系统时,我们需要区分两种架构:

### Workflow (工作流)
- 通过**预定义的代码路径**编排 LLM 和工具
- 适合任务明确、步骤固定的场景
- 可预测性强,易于调试

### Agent (智能代理)
- LLM **动态控制**自己的流程和工具使用
- 适合需要灵活决策的复杂任务
- 自主性强,但成本和复杂度更高

## 何时使用 Agent?

**使用 Agent 的场景:**
- ✅ 任务步骤无法预测
- ✅ 需要根据环境动态调整策略
- ✅ 可以容忍更高的延迟和成本
- ✅ 有明确的成功评估标准

**不需要 Agent 的场景:**
- ❌ 简单的单次 LLM 调用就能解决
- ❌ 任务流程完全固定
- ❌ 对成本和延迟要求极高
- ❌ 缺乏有效的反馈机制

**原则: 从最简单的方案开始,只在必要时增加复杂度。**

## 构建 Agent 的核心模式

### 1. 增强型 LLM (基础构建块)

这是所有 Agent 系统的基础:

```
LLM + 检索(Retrieval) + 工具(Tools) + 记忆(Memory)
```

**关键点:**
- 为 LLM 提供清晰的工具文档
- 确保工具接口易于理解和使用
- 可以使用 Model Context Protocol (MCP) 集成第三方工具

### 2. Prompt Chaining (提示链)

将任务分解为一系列步骤,每个 LLM 调用处理前一个的输出。

```
输入 → LLM₁ → 检查 → LLM₂ → 检查 → ... → 输出
```

**适用场景:**
- 生成营销文案,然后翻译成其他语言
- 先写大纲,检查后再写完整文档

**优势:** 用延迟换取准确性,每个步骤更简单

### 3. Routing (路由)

根据输入分类,路由到专门的处理流程。

```
输入 → 分类器 → 路由到专门的 LLM/流程
```

**适用场景:**
- 客服系统:将不同类型的问题路由到不同处理流程
- 模型选择:简单问题用小模型,复杂问题用大模型

**优势:** 关注点分离,针对性优化

### 4. Parallelization (并行化)

同时处理任务的不同部分,然后聚合结果。

**两种变体:**

**Sectioning (分段):** 将任务分成独立的子任务并行执行
```
输入 → [LLM₁, LLM₂, LLM₃] → 聚合 → 输出
```

**Voting (投票):** 多次运行同一任务,获取多样化输出
```
输入 → [LLM, LLM, LLM] → 投票/聚合 → 输出
```

**适用场景:**
- Sectioning: 内容审核(一个处理查询,一个检查安全性)
- Voting: 代码漏洞检测(多个角度评估)

### 5. Orchestrator-Workers (编排者-工作者)

中央 LLM 动态分解任务,委派给工作 LLM,并综合结果。

```
输入 → 编排者 → [工作者₁, 工作者₂, ...] → 编排者 → 输出
```

**适用场景:**
- 复杂代码修改(需要修改多个文件)
- 多源信息搜索和分析

**关键区别:** 子任务不是预定义的,而是由编排者根据输入动态决定

### 6. Evaluator-Optimizer (评估者-优化者)

一个 LLM 生成响应,另一个提供评估和反馈,循环迭代。

```
输入 → 生成器 → 评估器 → 反馈 → 生成器 → ... → 输出
```

**适用场景:**
- 文学翻译(需要多轮优化细节)
- 复杂搜索任务(评估器决定是否需要进一步搜索)

**判断标准:**
- LLM 响应能通过人类反馈明显改进
- LLM 能够提供有价值的评估反馈

### 7. Autonomous Agent (自主代理)

真正的 Agent:在循环中使用工具,根据环境反馈自主决策。

```
任务 → 规划 → 执行 → 观察结果 → 调整计划 → 执行 → ...
```

**核心特征:**
- 从用户命令或交互开始
- 独立规划和执行
- 在检查点或遇到障碍时返回人类
- 从环境获取"真实反馈"(工具调用结果、代码执行等)

**适用场景:**
- 解决 GitHub issue (SWE-bench)
- 计算机操作任务 (Computer Use)

**注意事项:**
- 成本更高
- 可能出现复合错误
- 需要在沙箱环境中充分测试
- 必须设置适当的护栏

## 工具设计的最佳实践

工具设计和提示工程同样重要!

### 1. 选择合适的工具格式

**原则:**
- 给模型足够的 token 来"思考"
- 保持格式接近互联网上自然出现的文本
- 避免格式"开销"(如精确计数代码行数)

**示例:**
- ✅ Markdown 中的代码块
- ❌ JSON 中的代码(需要转义换行符和引号)

### 2. 优化工具文档

把自己想象成模型:

- 📝 **清晰的描述**: 像给初级开发者写文档一样
- 📋 **包含示例**: 展示使用方法和边界情况
- 🎯 **明确参数**: 参数名和描述要一目了然
- 🔍 **区分相似工具**: 明确不同工具的使用场景

### 3. 防错设计 (Poka-yoke)

改变参数设计,让错误更难发生。

**案例:** SWE-bench Agent 开发中
- ❌ 问题: 模型在移出根目录后,相对路径工具出错
- ✅ 解决: 改为始终要求绝对路径
- 结果: 模型使用完美无误

### 4. 迭代测试

- 在 Workbench 中运行大量示例输入
- 观察模型犯的错误
- 持续迭代优化

## 实践中的 Agent 应用

### A. 客户支持

**为什么适合 Agent:**
- 自然的对话流程 + 需要访问外部信息
- 可集成工具拉取客户数据、订单历史
- 可编程处理退款、更新工单等操作
- 成功可通过用户定义的解决方案明确衡量

**商业验证:** 一些公司采用基于使用的定价模式,仅对成功解决的问题收费

### B. 编程助手

**为什么适合 Agent:**
- 代码解决方案可通过自动化测试验证
- Agent 可使用测试结果作为反馈迭代
- 问题空间定义明确且结构化
- 输出质量可客观衡量

**实际案例:** Anthropic 的 Agent 可以仅根据 PR 描述解决 SWE-bench Verified 中的真实 GitHub issue

## 核心原则总结

构建成功的 Agent 系统,遵循三个核心原则:

### 1. 保持简单
- 从简单提示开始
- 通过全面评估优化
- 仅在简单方案不足时添加多步骤系统

### 2. 优先透明
- 明确展示 Agent 的规划步骤
- 让决策过程可观察、可调试

### 3. 精心设计 ACI (Agent-Computer Interface)
- 像重视 HCI 一样重视 ACI
- 通过详尽的工具文档和测试优化接口

## 框架的使用建议

**常见框架:**
- Claude Agent SDK
- AWS Strands Agents SDK
- Rivet (拖拽式 GUI 工具)
- Vellum (GUI 工具)

**建议:**
- 🚀 **快速开始**: 框架简化标准任务
- ⚠️ **注意抽象**: 可能掩盖底层提示和响应,难以调试
- 💡 **先用 API**: 许多模式几行代码就能实现
- 🔍 **理解底层**: 如果使用框架,确保理解底层代码

## 下一步行动

1. **评估需求**: 你的任务真的需要 Agent 吗?
2. **选择模式**: 从最简单的适用模式开始
3. **设计工具**: 投入时间优化工具接口
4. **迭代测试**: 在沙箱环境中充分测试
5. **监控优化**: 持续测量性能并改进

记住:**成功不在于构建最复杂的系统,而在于构建最适合需求的系统。**
